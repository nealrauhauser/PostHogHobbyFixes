# PostHog Hobby Deploy — Issue Registry

**Date: 2026-04-05** | **Host: pilot (pilotvps)** | **Stack: PostHog self-hosted hobby**

Every issue encountered deploying PostHog hobby, in discovery order. All fixes are automated in `setup.sh`. See also `PostHogDeployHowTo.md` for the step-by-step deploy guide.

**TL;DR**: PostHog hobby deploy is broken out of the box. The upstream compose assumes cloud infra (managed Redis with TLS, specific binary paths, correct env propagation). Our `setup.sh` + `docker-compose.override.yml` + `dev-services.env` fix everything. If you're here because something broke after a `docker compose pull`, jump to the [Post-Update Checklist](#post-update-checklist) at the bottom.

## Issue 1: ClickHouse Projection Mode

**Error**: `Code: 344. Projections are not supported for ReplacingMergeTree with deduplicate_merge_projection_mode = throw`

**Cause**: ClickHouse 24.8+ changed default from `drop` to `throw`. PostHog migrations create projections on ReplacingMergeTree tables.

**Fix**: Add `<deduplicate_merge_projection_mode>drop</deduplicate_merge_projection_mode>` to `default.xml`.

---

## Issue 2: KafkaEngine DEFAULT Columns

**Error**: `Code: 36. KafkaEngine doesn't support DEFAULT/MATERIALIZED/EPHEMERAL expressions for columns`

**Cause**: `person_sql.py` shares SQL templates between MergeTree and Kafka tables. Two columns have `DEFAULT` clauses banned in Kafka engine since ClickHouse 23.3.

**Fix**: Patch `person_sql.py`, mount via Docker volume. Upstream: [PostHog #15625](https://github.com/PostHog/posthog/issues/15625).

**Trap**: Migration runner records "applied" even when operations fail. Registry gets ahead of reality — requires full data wipe to recover.

---

## Issue 3: Caddy Proxy Binding

**Cause**: `CADDY_HOST` in base compose resolves at compose-parse time, not runtime. Override value arrives too late.

**Fix**: Set `CADDY_HOST=http://:80` in `.env` (compose-parse time).

---

## Issue 4: Plugins Fernet Key

**Cause**: Plugins service requires `ENCRYPTION_SALT_KEYS` as 32-char base64url (24 random bytes). Bootstrap generates wrong format. Hobby compose doesn't pass it to the plugins container.

**Fix**: Generate proper key, set in `.env` and override.

---

## Issue 5: Override File Not Loading

**Cause**: `docker compose -f docker-compose.hobby.yml` silently disables auto-loading of `docker-compose.override.yml`. All overrides ignored.

**Fix**: `COMPOSE_FILE=docker-compose.hobby.yml:docker-compose.override.yml` in `.env`.

## Issue 6: Node Services Redis — Wrong Host

**Error**: `😡 [recording-api] Redis error encountered! host: 127.0.0.1:6379 Enough of this, I quit!`

**Cause**: Each Node sub-service defines its own `*_REDIS_HOST` env var defaulting to `127.0.0.1`. Setting `REDIS_URL` alone is insufficient — five vars must be set independently:

| Env Var | Default | Override |
|---------|---------|----------|
| `REDIS_URL` | `redis://127.0.0.1` | `redis://redis7:6379/` |
| `CDP_REDIS_HOST` | `127.0.0.1` | `redis7` |
| `LOGS_REDIS_HOST` | `127.0.0.1` | `redis7` |
| `TRACES_REDIS_HOST` | `127.0.0.1` | `redis7` |
| `SESSION_RECORDING_API_REDIS_HOST` | `127.0.0.1` | `redis7` |

**Fix**: All five in `dev-services.env`, loaded via `env_file:` in override.

## Issue 7: Node Services Redis — TLS Handshake Timeout

**Error**: `ETIMEDOUT` connecting to `redis7:6379` (correct host, correct port, TCP connectivity works)

**Cause**: `isProdEnv()` returns `true` inside Docker (image ships with `NODE_ENV=production`). This enables TLS on Redis connections by default. Plain Redis doesn't respond to TLS ClientHello — handshake hangs until TCP timeout.

**Affected vars**:

| Env Var | Default (in "prod") |
|---------|---------------------|
| `LOGS_REDIS_TLS` | `true` |
| `TRACES_REDIS_TLS` | `true` |

**Fix**: Set `LOGS_REDIS_TLS=false` and `TRACES_REDIS_TLS=false` in `dev-services.env`.

**Trap**: The original Issue 6 fix only added `env_file: dev-services.env` to the `plugins` service. PostHog hobby runs separate containers for each ingestion workload — all need the env_file:

- `plugins`
- `ingestion-general`
- `ingestion-logs`
- `ingestion-traces`
- `ingestion-sessionreplay`
- `recording-api`

## Issue 8: ingestion-general Invalid PLUGIN_SERVER_MODE

**Error**: `Error: Invalid PLUGIN_SERVER_MODE ingestion-v2-combined`

**Cause**: PostHog removed the `ingestion-v2-combined` mode in a recent release. The hobby compose still references it.

**Valid modes** (as of `posthog/posthog-node:latest` pulled 2026-04-05):
`ingestion-v2`, `ingestion-logs`, `ingestion-traces`, `ingestion-errortracking`, `recording-api`, `local-cdp`, `cdp-*` variants, `evaluation-scheduler`

**Fix**: Override in `docker-compose.override.yml`:
```yaml
  ingestion-general:
    environment:
      PLUGIN_SERVER_MODE: "ingestion-v2"
```

## Issue 9: temporal-django-worker Exit 127

**Error**: `./bin/temporal-django-worker: No such file or directory`

**Cause**: The `temporal-django-worker` binary was removed from the `posthog/posthog` image, but the hobby compose still defines the service and references the binary. Not in `./bin/`, not as a Django management command — fully gone.

**Fix**: Override the entrypoint with a no-op in `docker-compose.override.yml`:
```yaml
  temporal-django-worker:
    entrypoint: ["echo", "temporal-django-worker disabled — binary removed from image"]
```

**Impact**: None for hobby deploy. Temporal handles async exports and batch operations, not core event ingestion or analytics.

## Issue 10: Validation Page "Plugin server - Error" (Stale Health Check)

**Error**: Validation page shows "Plugin server · Node — Error" despite the plugin server being fully healthy.

**Cause**: The Django web service checks plugin server health by reading `@posthog-plugin-server/ping` from Redis (`posthog/utils.py:725`). The plugin server is supposed to write a timestamp to this key every few seconds. **The Node plugin server no longer writes this key** — the heartbeat code was removed from the image. The `/_health` endpoint on port 8001 returns `status: ok` with all 20 checks passing, but the validation page doesn't use that endpoint.

**Proof**:
```bash
# Health endpoint is fine
curl -s http://plugins:8001/_health  # → {"status":"ok","checks":{...all ok...}}

# But the Redis key is missing
redis-cli GET "@posthog-plugin-server/ping"  # → (nil)

# Heartbeat writer doesn't exist in the image
grep -rn 'plugin-server/ping' /code/nodejs/dist/ --include='*.js'  # → nothing
```

**Fix**: Seed the key manually and keep it alive. Add to `docker-compose.override.yml`:
```yaml
  # Workaround: plugin server no longer writes heartbeat to Redis, but
  # Django validation page still checks for it. This sidecar writes the
  # key every 10 seconds so the health check passes.
  plugin-server-heartbeat:
    image: redis:7-alpine
    entrypoint: sh
    command: -c 'while true; do redis-cli -h redis7 SET "@posthog-plugin-server/ping" "$$(date -u +%Y-%m-%dT%H:%M:%S.000Z)" EX 30; redis-cli -h redis7 SET "@posthog-plugin-server/version" "unknown"; sleep 10; done'
    depends_on:
      - redis7
    restart: unless-stopped
```

**Also**: The plugin server moved its HTTP health port from `8001` to `6738` (`DEFAULT_HTTP_SERVER_PORT` in `common/config.js`). The hobby compose still declares `8001/tcp`. Fix: set `HTTP_SERVER_PORT=8001` in `dev-services.env`. Without this, the health endpoint works but on the wrong port.

**Impact**: Cosmetic only — the plugin server is fully operational. This workaround unblocks the validation page so you can proceed with setup.

## Pattern

PostHog hobby deploy is a second-class citizen. The upstream compose assumes cloud infra (managed Redis with TLS, specific ClickHouse versions, correct env propagation). Every `docker pull` risks new breakage because:

1. **Fragmented config** — each Node sub-service defines its own env vars with localhost defaults
2. **isProdEnv() gate** — Docker images ship as "production", enabling TLS/auth features that hobby Redis doesn't support
3. **Mode enum churn** — `PLUGIN_SERVER_MODE` values get renamed/removed without hobby compose updates
4. **Silent failures** — migrations record success on partial failure, env vars silently fall back to defaults

## Issue 11: posthog_team Missing project_id Column

**Error**: `column t.project_id does not exist` — crashes `ingestion-general` and `plugins` in a loop.

**Cause**: PostHog's Node ingestion code now queries `SELECT t.id, t.project_id, ... FROM posthog_team t` but the Django migration that adds `project_id` to `posthog_team` hasn't shipped in the hobby release. The column is also expected on `posthog_grouptypemapping`.

**Fix**: Add the column manually, mirroring team `id`:
```sql
-- posthog_team
ALTER TABLE posthog_team ADD COLUMN project_id bigint;
UPDATE posthog_team SET project_id = id;
ALTER TABLE posthog_team ALTER COLUMN project_id SET NOT NULL;

-- posthog_grouptypemapping
ALTER TABLE posthog_grouptypemapping ADD COLUMN project_id integer;
UPDATE posthog_grouptypemapping SET project_id = team_id;
ALTER TABLE posthog_grouptypemapping ALTER COLUMN project_id SET NOT NULL;
```

Then restart `ingestion-general` and `plugins`.

**Impact**: Critical — without this, no events are ingested. The capture API accepts events into Kafka but the ingestion pipeline can't process them into ClickHouse.

## Post-Update Checklist

Run this after every `docker compose pull` or PostHog version bump. Each command checks for a class of breakage that PostHog has introduced before and will introduce again.

```bash
cd ~/posthog

# ── 1. Check for new 127.0.0.1 Redis defaults ──────────────────────
# If new vars appear, add them to dev-services.env pointing to redis7
docker compose run --rm plugins grep -rn '127\.0\.0\.1' /code/nodejs/dist --include='config.js'

# ── 2. Check for new TLS defaults ──────────────────────────────────
# If new *_REDIS_TLS vars appear, add them as =false to dev-services.env
docker compose run --rm plugins grep -rn 'REDIS_TLS' /code/nodejs/dist --include='config.js'

# ── 3. Check valid PLUGIN_SERVER_MODE values ───────────────────────
# If ingestion-general crashes with "Invalid PLUGIN_SERVER_MODE", the
# mode was renamed. Find the new name here and update the override.
docker compose run --rm plugins node -e "const t = require('/code/nodejs/dist/types'); console.log(Object.keys(t.stringToPluginServerMode))"

# ── 4. Check health port hasn't moved again ────────────────────────
# If validation page shows plugin server error but container is running,
# check what port it's listening on. Should match HTTP_SERVER_PORT in dev-services.env.
docker compose run --rm plugins grep -n 'DEFAULT_HTTP_SERVER_PORT' /code/nodejs/dist/common/config.js

# ── 5. Check if temporal-django-worker binary reappeared ───────────
# If it exists again in a future image, we can re-enable the service.
docker compose run --rm temporal-django-worker sh -c "ls ./bin/ | grep temporal" 2>/dev/null || echo "Still missing — keep disabled"

# ── 6. Check for new Node services needing env_file ────────────────
# Any new service using the posthog-node image needs env_file: dev-services.env
# in the override, or it will default to 127.0.0.1 Redis with TLS.
docker compose config --services | while read svc; do
  img=$(docker compose config --format json | python3 -c "import sys,json; svcs=json.load(sys.stdin)['services']; print(svcs.get('$svc',{}).get('image',''))" 2>/dev/null)
  if echo "$img" | grep -q "posthog-node\|posthog/posthog-node"; then
    has_env=$(docker compose config --format json | python3 -c "import sys,json; svcs=json.load(sys.stdin)['services']; ef=svcs.get('$svc',{}).get('env_file',[]); print('yes' if ef else 'no')" 2>/dev/null)
    [ "$has_env" = "no" ] && echo "WARNING: $svc uses posthog-node image but has no env_file"
  fi
done

# ── 7. Verify everything is up ─────────────────────────────────────
docker compose ps -a --format "table {{.Name}}\t{{.Status}}" | grep -v "Up \|Exited (0)"
```
